     # Setup Guide for Windows with Docker

     This guide helps you set up the Vape Goblin Town project on Windows using Docker for SpacetimeDB.

     ## Prerequisites

     - Docker Desktop for Windows (installed and running)
     - Node.js v22+ (already installed)
     - npm (already installed)
     - Git (to clone the repo)

     ## Setup Steps

     ### 1. Start SpacetimeDB with Docker

     From the project root directory, start SpacetimeDB using docker-compose:

     ```bash
     docker-compose up -d spacetimedb
     ```

     This will:
     - Download the SpacetimeDB Docker image (if not already present)
     - Start a SpacetimeDB server running on `localhost:3000`
     - Create a persistent volume for data

     Verify it's running:
     ```bash
     docker ps | grep vape-goblin-spacetime
     ```

     ### 2. Build the Server Code

     Since SpacetimeDB CLI is not available on Windows natively, we need to use Docker to build the server. Run this
      command from the project root:

     ```bash
     docker run --rm -v "${PWD}:/app" -w /app/server clockworklabs/spacetime:latest spacetime build
     ```

     This will:
     - Use the SpacetimeDB Docker image to compile your Rust server code
     - Generate the WASM module in `/server/target/`

     ### 3. Generate TypeScript Bindings

     Still from the project root:

     ```bash
     docker run --rm -v "${PWD}:/app" -w /app/server clockworklabs/spacetime:latest spacetime generate --lang
     typescript --out-dir ../client/src/generated
     ```

     This will:
     - Generate TypeScript client bindings from your Rust server schema
     - Place them in `client/src/generated/`

     ### 4. Install Client Dependencies

     ```bash
     cd client
     npm install
     cd ..
     ```

     ### 5. Publish Your Module to SpacetimeDB

     ```bash
     docker run --rm --network host -v "${PWD}:/app" -w /app/server clockworklabs/spacetime:latest spacetime publish
      vibe-multiplayer
     ```

     This:
     - Connects to your running SpacetimeDB server
     - Publishes your module to it
     - Uses `--network host` to allow container to reach `localhost:3000`

     ### 6. Start the Development Client

     ```bash
     cd client
     npm run dev
     ```

     The client will be available at `http://localhost:5173`

     ## Development Workflow

     When you modify server code (schema, reducers, etc.):

     1. Rebuild: `docker run --rm -v "${PWD}:/app" -w /app/server clockworklabs/spacetime:latest spacetime build`
     2. Regenerate bindings: `docker run --rm -v "${PWD}:/app" -w /app/server clockworklabs/spacetime:latest
     spacetime generate --lang typescript --out-dir ../client/src/generated`
     3. Republish: `docker run --rm --network host -v "${PWD}:/app" -w /app/server clockworklabs/spacetime:latest
     spacetime publish vibe-multiplayer`

     ## Stopping SpacetimeDB

     ```bash
     docker-compose down
     ```

     To also remove the persistent data volume:

     ```bash
     docker-compose down -v
     ```

     ## Troubleshooting

     ### "Cannot connect to localhost:3000"
     - Make sure `docker-compose up -d spacetimedb` is running
     - Check status: `docker ps`

     ### Permission denied errors on Windows PowerShell
     - Use cmd.exe instead, or
     - Escape the paths differently for PowerShell

     ### Generated files not showing up
     - Verify the docker command completed successfully (check the output)
     - The files should appear in `client/src/generated/`

     ## Notes

     - The SpacetimeDB server persists data in a Docker volume named `spacetime_data`
     - All docker commands use volume mounts to access your local files
     - The `--network host` flag in the publish command allows the container to reach your local SpacetimeDB server

  mounts
  - Browser couldn't connect back to services on host machine

  4. SpacetimeDB Cloud Publishing ✅ (What Actually Worked!)

  This worked because:
  - No native Windows binary needed - just the CLI for publishing
  - Cloud handles all the database/server infrastructure
  - Client just connects via WebSocket - platform agnostic
  - React dev server runs locally on Windows (no Docker needed for dev)

  The Winning Architecture

  Your Windows Machine
  ├─ npm run dev → Vite server (localhost:5173)
  │  └─ React/Three.js rendering in browser
  │
  └─ SpacetimeDB CLI (installed on Windows)
     └─ Publishes to SpacetimeDB Cloud
        └─ Browser connects via wss://maincloud.spacetimedb.com
           ↓
           SpacetimeDB Cloud (Hosted)
           └─ Database + Server logic

  Key Lessons

  1. Platform doesn't matter for the client - React runs anywhere (Windows,
  Mac, Linux, browser)
  2. Backend as a service > self-hosted - Avoiding the complexity of running
   SpacetimeDB locally was the real win
  3. WebSocket over HTTPS - Client-server communication is language/OS
  agnostic
  4. Version mismatches are real - CLI 1.9.0 generating code for SDK 1.3.3
  was brutal, but we worked around it

  You basically proved that cloud-first development beats fighting with
  local infrastructure issues!
